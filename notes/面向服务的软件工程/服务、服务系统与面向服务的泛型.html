# 服务、服务系统与面向服务的泛型

# 服务

# 服务与货物

服务：无形的、非持久的体验。

服务发生在客户、雇员/资源、系统（通常是IT系统）之间的交互中。

货物：有形的、可存储的产品。

事实上，没有实体的东西大部分是服务。包括咨询、管理、商务，都属于服务的范畴。

# 服务模式与制造模式

# 区别

服务模式：双向交互（消费者参与，可定制，用户反馈快），产物是服务（或以物体为载体的服务）。

制造模式：关系不明确（消费者不参与，批量生产不可定制，用户反馈慢），产物是货物。

制造模式的产物一定是有形的，而服务模式的产物不一定是无形的；主要的区别还是在定制和用户参与上。

# 融合

服务与制造的融合，体现在向制造中引入服务，增加用户的选择。

Tips

  • 汽车：制造模式
  • 不同型号的汽车：制造模式（融合了服务特征，增加了选择；但本质还是制造模式，因为用户并没有参与生产过程，只是被动地在给定的选项中进行选择）。
  • 高端手工制作汽车：服务模式（以物体为载体的服务，虽然产物是物体，但是具有定制化的特征。想想，买这种车的人真的是为了买车吗？）
    • 插一句，据说劳斯莱斯幻影是手工制作的，但为啥我觉得这车好丑……
    • 再插一句，到目前为止我觉得手感最好的车是桑塔纳（认真）。
  • 根据用户反馈进行改进的限量款耳机：制造模式（虽然根据用户反馈进行了改进，并且限量款也具有服务的一些特征，但本质上用户没有参与，而且拿到手里的耳机其实还是一样的，并没有定制化）
  • 定制衣服：服务模式，同高端手工汽车。买的就是定（xuan）制（yao）。
  • 定制家具：服务模式，同定制衣服。

简而言之，基本上只要是定制的，就是服务模式。

# 服务系统

业务服务：能够满足一般消费者对于某一方面特定需求的服务。

服务系统：用于实现业务服务的IT系统。

IT系统其实就是软件系统，换个说法而已。

事实上，只有少数的IT系统不是服务系统。但仍然有一些IT系统不是（或不完全是）服务系统，比如操作系统、word。

个人认为，服务的主要特征是能够独立完成某一个业务功能（满足自治），并且要通过网络对外发布（满足开放、网络组件）。

操作系统并不能完成一个业务功能，只是一个基础支撑。我们知道OS的目的是管理系统资源和给提供用户操作系统资源的接口，这些都不属于业务功能。

word这种软件是不自治的，虽然能完成业务功能，但并不能独立运行；而且也不是通过网络对外发布的。这玩意感觉怎么都解释得通，word是目的是为了编辑文件，写出来的东西还能提供给其他软件使用，数据上就不隔离，破坏了自治性。

所以服务会特别强调独立的计算结点，因为只有独立的机器（或者虚拟机，或者容器）才能真正隔离。哪怕做不到真正的隔离，也要在概念上隔离。

# IT服务与非IT服务

如果业务服务由服务系统提供，则该服务称为IT服务。

IT服务又称IT使能（IT-enabled）服务，这只是一个名词上的trick而已。

简单来说，只要有服务系统参与的业务服务，就是IT服务；里面可能既有IT服务，又有非IT服务。

比如医院看病的整个过程，医院为了提供医疗服务，会有大量的服务系统支撑，比如挂号、收费服务，就是IT服务；但真正看病的还是人，看诊服务还是非IT服务。

完全没有IT系统参与的服务才是非IT服务。

# 区别

  • KPIs不同：IT服务有统一的度量（如性能、吞吐率），非IT服务没有统一的度量
  • 需求管理不同：IT服务的需求容易量化，非IT服务的需求不容易量化，管理方式不同
  • 变化的步调不同：IT服务不容易变更，非IT服务容易变更

# 问题

如何有效地构建、维护服务系统？

  • 复杂度高
  • 需求变更多
  • 计算环境不断演化
  • 内部完成还是外包？
  • IT专家和业务专家的沟通困难
  • 功能相近的服务簇

# 面向服务的泛型

泛型：促进复用，减少无用开发。

这里的泛型本质是复用。

# 面向过程的泛型

复用过程（计算逻辑），无法复用状态。

# 面向对象的泛型

通过封装、继承、多态来复用状态，以及复用对状态的操作和消息传递。

面向对象是不稳定的，容易受到外界因素的影响，即使是使用了设计模式，也只是缓解问题，并不能改变在发生变更时要重新设计对象这一事实；因为面向对象的本质还是利用对象间的继承实现对计算的复用。

面向对象的实质是对现实世界的实体的建模，所以一旦实体发生变化，面向对象模型就一定会发生变化。

# 基于构件的泛型

构件：模块化、可部署、可替换的功能组件，对内封装实现，对外提供标准化接口。

以功能为单位，通过组合进行复用，变更被控制在构件内部。

构件开发和具体业务分离：

  • 构件开发：构建的创建（领域分析、领域设计、领域实现）、管理
  • 具体业务：复用现有的构件形成业务流程，分析、设计、实现、测试

构件内部可以是面向对象的，也可以是别的任何形式；我们并不关心它的内部实现，只要对外提供的功能不变即可。

# 面向服务的泛型

服务 ：开放、自治、自描述的网络构件（满足四个约束的构件）。

与基于构件的泛型相比，复用范围更广，因为服务是开放的，可以跨服务系统进行复用。

事实上，在服务之前，其他三种泛型都是对代码的复用，而服务则超越了这个范畴，开始对计算能力进行复用。构件本身是对某个计算功能的复用，但它只能用于本地，还是得把组件的代码放进自己的代码才能发挥作用（比如组件库）。

基本活动仍然是构件的开发、管理与具体业务的实现。
