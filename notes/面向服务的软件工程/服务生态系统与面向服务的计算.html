# 服务生态系统与面向服务的计算

# 服务生态系统

# 服务组合

顾名思义，将多个服务组合为一个服务，以完成业务任务。

其实就是一个组合模式。

# 服务库存

服务库存：在组织或组织的合理边界内，一组独立标准化并治理的服务的集合。

在服务库存构建前，应该有一个蓝图。

# 内容

服务库存分为三层：

  • 应用服务层：满足开发者的需求

    对单个功能进行抽象（如打印服务），避免重复开发，降低（其他人的）开发难度。

    个人感觉，一般来说应用服务不会是一个流程，他应该是整个流程中的一个步骤。比如这个打印服务，就只是为了打印这一个步骤而已。

  • 业务服务层：满足用户的需求

    • 任务服务：对流程进行抽象（如报道服务，包括了整个报道的全部流程）
    • 实体服务：对实体进行抽象（如作业服务，提交的流程可能各不相同，但最终都是对作业这个实体进行的操作）

    可以用单个服务实现，也可以用服务组合实现。

  • 编排服务层（可选）：满足业务专家的需求

    • 可以看作是服务组合的一种实现
    • 利用平台中立的语言描述服务组合，描述业务逻辑。

    主要是业务专家未必会写代码，他需要一些简单的方式来描述他想要的业务流程。

# 演化

“龙卷风模型”：随着时间推移，服务库存逐渐完善，可复用的服务逐渐增加，开发的成本逐渐降低。

按照蓝图逐渐扩展服务库存，优先开发可能用到的服务（按需开发），并且在开发时从库存中寻找可复用的服务。

# 服务生态系统

当组织的服务库存已经基本完备，服务系统已经合理地转向面向服务的实现时，服务生态系统就完成了构建。

合理指的是不是所有系统都需要转向面向服务的，比如OS。

包括分析、设计、实现、治理、演化等过程。

治理：因为服务是开放的网络构件，需要面向所有其他人，如何提供足够的计算资源？

演化：服务的质量属性，或者干脆功能已经不能满足需求的时候，如何应对？演化过程中，接口兼容还是不兼容？

# 水平服务和垂直服务

这个是针对服务消费者的概念。

垂直服务：可以被同时、独立调用以满足消费者需求的服务。

水平服务：构成垂直服务的多个可重用的跨领域的公共服务。

说白了就是构成垂直服务的服务组合中的那些服务。

并且这两者的定义不互斥，在不同的场景下，同一个服务可能既是水平服务，又是垂直服务。举例来说，如果我只是想发一封邮件，那邮件服务就是垂直服务；如果我想去报道，而发邮件是其中的一个步骤，那邮件服务其实就是水平服务。

水平服务、垂直服务与应用服务、业务服务、编排服务是两种不同的分类方式，没有什么必然联系。

# 面向服务的计算

# 定义

泛型角度：SOC，用服务对计算进行抽象和复用，以实现快速、低成本开发，分布式异构环境的组合

架构角度：SOA，使用标准接口的独立构件来进行构造

服务角度：在业务服务和IT服务之间建立联系（对领域的分析、设计、实现，对服务的管理）

软件工程角度：以服务为基本抽象单位，进行分析、设计、开发、测试、部署、维护、管理

# 目标

# 技术

  • 灵活性
  • 可靠性
  • 可扩展性
  • 业务和技术的一致性（与商业一致）
  • 可复用性（降低成本）

以及由此带来的高生产率。

# 商业

  • 供应商多元化
  • 互操作性
  • 组织结构的灵活性
  • 商业联盟
  • 可复用性（降低成本）
  • 业务和技术的一致性（与技术一致）
  • ROI投资回报率的提升

# 面向服务和面向对象的差异

# 总体

方法论、协作、代码复用、动态绑定、重组、通信方式、系统维护（可用性）、可靠性、软件拥有方式

特点 面向对象 面向服务
方法论 通过紧耦合的类进行应用开发
基于继承的层次式架构
自底向上的构造函数 - 类和模型 - 系统设计		 通过松耦合的服务进行应用开发
自顶向下的系统模型 - 服务模块，服务抽象 - 服务实现
通过搜索获得可用的服务实现（服务发现）
抽象和协作层次 一个团队负责应用开发，负责整个生命周期
开发者必须了解应用领域知识和编程		 开发任务由三个独立方承担：应用程序开发者，服务提供方和服务代理
应用程序开发者了解应用逻辑，但不关心服务的实现
服务提供者需要编程能力，但不关心服务的应用
服务代理作为中介，实现解耦
代码共享和复用 继承和库函数
编译时引入
往往是平台相关的		 服务
平台无关，能够被远程调用
支持系统的服务共享
动态绑定和重新组合 运行时将名称与方法关联
方法必须在应用部署前链接到可执行的代码		 运行时将服务调用和服务进行绑定
可以在应用部署后进行服务选择
可以在运行时重组
重组 在设计时决定导入的组件 可以动态改变服务的组合关系
组件通讯和接口 与平台和语言有关 与平台和语言无关，组件间通过标准协议通信
系统维护 用户自行升级
升级时应用必须停止		 通过互联网升级
维护对用户透明
可靠性 设计时决定 服务本身相对简单，本身更加可靠
可以进行动态的服务间切换，来确保可靠
软件拥有 作为产品销售，为用户所有 用户每次对服务使用付费，而不是按照产品付费

# 设计

耦合、粒度、作用域、前瞻性（对未来变更的预测）、状态性、组合方式

特点 面向对象 面向服务
耦合 提倡重用和松耦合
但是预先定义的类依赖导致对象紧耦合		 松耦合
由功能和服务合约给定
粒度 支持不同规模的任务
支持细粒度API		 鼓励粗粒度API（服务描述）
通信消息中包括尽可能多的任务相关信息
作用域 小，基于特定的软件系统 大，基于服务生态系统
前瞻性 鼓励逻辑与数据绑定产生对象
逻辑修改时，对象需要修改		 鼓励创建活动无关的、消息驱动的服务
功能抽象，功能不易变更
状态性 数据和逻辑绑定，带状态的对象
相对不易预测		 尽可能无状态的服务
容易预测
组合 对象继承导致紧耦合 服务组合的松耦合

不一而足。

我们有面向对象的语言，但我们永远也不会有面向服务的语言；因为服务本身是独立于语言的，从概念上就是矛盾的。
