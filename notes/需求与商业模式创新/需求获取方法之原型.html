# 需求获取方法之原型

# 不确定性与风险

不确定性：因为对未来知识了解有限而无法确定某些行为或事件的后果。

解决不确定性：

  1. 增加对未来知识的了解程度，如天气预报。
  2. 风险管理，如带上雨伞。

风险：因为不确定性而可能给未来造成的损失。

# 原型

如果在最终制品产生之前，一个中间制品被用来在一定广度和深度范围内表现这个最终制品，那么这个中间制品就被认为是最终制品在该广度和深度上的原型。

原型是一个系统，它内化（capture，或者译为捕获？）了一个更迟系统的本质特征。

原型的核心是通过将未来的知识带回现在，以解决不确定性。

# 分类

# 用途

不确定性存在于不同的开发阶段（项目启动？需求？开发？后续迭代？）。

  1. 演示原型：项目启动阶段，展示技术实力
  2. 严格意义上的原型：需求阶段，解决需求的不确定性
  3. 试验原型：开发阶段，解决技术的不确定性（某项技术是否可行等）
  4. 引示系统原型：作为进一步迭代、演化的基础

# 开发方法

不确定性存在于不同的位置（需求？技术？演化？）

  1. 探索式：需求不明确
  2. 实验式：需求明确，但使用什么技术不明确
  3. 演化式：作为进一步演化的资产

# 是否抛弃

  1. 抛弃式：包括探索式和实验式，可以比较草率
  2. 演化式：需要精心设计以确保高质量，要符合目标系统的约束

# 过程

  1. 确定原型需求：为什么，什么时候开始，什么时候结束？
  2. 原型开发：成本尽可能低
  3. 原型评估：是否满足结束标准？
  4. 原型修正

# 成本

要坚决抛弃抛弃式原型。

# 水平原型

仅仅实现某些层次的功能（如用户界面层）。

主要的三个层次：

  • 人机交互，注重外观和交互方式
  • 功能与任务，注重提供哪些功能
  • 实现，注重实现上的细节。

# 垂直原型

实现特定功能的全部层次。

# 介质

# 低保真

纸面原型：成本低，但交互性不足。尽量使用。

幻灯动画：多个纸面原型的集成。

一般用于项目早期，不能充分获得用户反馈，但成本低。

# 高保真

原型工具：集中于用户界面的建立

程序语言：一般用于实验式和演化式的原型。

能够很好地获取用户反馈，但成本很高，且容易被用户与真实产品相混淆。

# 风险

  1. 成本失控：开发原型本身就是需要成本的。

    对于明确的需求没必要使用原型，使用原型时也要尽可能使用低成本的手段。

  2. 给客户造成错误印象：用户可能会把原型当成产品，从而提出不当要求，如加快交付。

  3. 可能会忽略功能：用户可能会关注原型的非功能因素，而忽略了功能是否完善。

  4. 可能会掩盖用户假设：用户可能会默认原型的一些特性也会在真实产品中体现出来（如性能），导致忽略这部分的需求。

# 故事板

通过组织静态原型来构建场景，类似于动画。

# 要素

角色（who）、内容（what）、方法（how）

# 类型

被动：给用户“讲”故事，偏静态（如图片、PPT，由展示者主导）

主动：让用户“看”故事，偏动态（如自动播放的PPT、视频，是一个比较中立的状态）

交互：让用户体验故事，用户主动进行探索
